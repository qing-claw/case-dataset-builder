# Case Dataset Builder：case 与项目说明

## 1. 这是什么项目

**Case Dataset Builder** 是一个面向**动漫文生图测试集制作**的轻量网页工具。

它的目标很直接：

- 把测试 case 写成统一结构
- 把 prompt、参考图、检查点、通过规则收进同一条记录里
- 让单条 case 可以独立导出、复用、分发
- 让多人协作时少一点“各写各的、各存各的”混乱

它不是评测模型本身的工具，也不是训练平台。
它更像是**测试集整理器 / case 编辑器**：先把题目出干净，再谈后续跑模型、打分、汇总结果。

在线地址：

- https://qing-claw.github.io/case-dataset-builder/

## 2. 什么是 case

在这个项目里，一个 **case** 可以理解为：

> 一条可独立使用、可重复执行、可被检查的测试样例。

它通常回答的是这样一个问题：

- 想测模型的哪种能力？
- 用什么 prompt 去触发？
- 是否需要参考图？
- 结果出来后重点看什么？
- 怎样才算通过？

所以一个 case 不是一句 prompt 那么简单。
它更完整，通常由下面几部分组成：

- **dimension**：测试维度
- **sub_dimension**：更细的子能力点
- **prompt**：实际输入给模型的提示词
- **ref**：参考图路径（如果该 case 依赖参考图）
- **check_points**：人工检查时重点观察的点
- **pass_rule**：什么情况下算通过

换句话说，**case 是一条结构化测试题**，而不是一段随手记下来的描述。

---

## 3. 为什么要把 case 结构化

如果没有统一结构，测试集很容易出现这些问题：

- prompt 写了，但不知道在测什么
- 有参考图，但图和 case 对不上
- 大家各自命名，后续脚本难处理
- 同一个能力点被重复写很多次，口径还不一致
- 结果回看时，只知道“感觉一般”，不知道该按什么标准判断

把 case 结构化之后，会得到几个很实际的好处：

### 3.1 单条 case 自包含
一条 case 自己就带着定义、参考图和规则，迁移、分发、归档都更方便。

### 3.2 便于多人协作
不同人可以分别补充 case，但最终仍然落到同一种格式里，方便合并。

### 3.3 便于后续自动化
统一结构后，后面无论要做：

- 批量运行模型
- 批量收集输出
- 自动生成评测面板
- 统计通过率
- 做 benchmark / 回归测试

都更容易接上。

### 3.4 降低“脏数据”概率
很多测试集不是缺数据，而是缺规矩。结构先统一，后面出问题会少很多。

## 4. 这个项目里的 case 长什么样

项目采用的是一个很直接的目录结构：

```text
dataset/
  cases/
    <case_folder>/
      case.json
      ref/
        ref_01.png
        ref_02.jpg
```

每条 case 对应一个独立文件夹，里面通常包含：

- `case.json`：该 case 的结构化定义
- `ref/`：参考图目录（如果有）

一个典型的 `case.json` 例子：

```json
{
  "dimension": ["风格表现"],
  "sub_dimension": ["指定画风还原"],
  "prompt": [
    "1girl, white hair, blue eyes, school uniform, anime style"
  ],
  "ref": ["ref/ref_01.png"],
  "check_points": ["画风接近", "线条干净", "上色方式一致"],
  "pass_rule": "3 个检查点满足 2 个及以上算通过"
}
```

这里有几个关键点：

### 4.1 `dimension`
表示大类能力，比如：

- 风格表现
- 角色表现
- 复杂构图能力
- 指令遵循
- 提示词适配能力
- 边界案例

它回答的是：**这条 case 大体在测什么。**

### 4.2 `sub_dimension`
表示更细的测试点，比如：

- 指定画风还原
- 角色换装遵循
- 多人物互动
- 数量约束遵循
- 中英文 prompt 适配
- 风格混合
- 极端姿势稳定性

它回答的是：**具体测的是哪个细分问题。**

### 4.3 `prompt`
这是实际给模型的输入内容。

这里设计成字符串列表，是因为同一个 case 可能会有：

- 多种等价说法
- 中英文版本
- 不同措辞强度的提示词

这样可以把“同一类测试意图”的多种 prompt 写在一起，而不是拆成很多零散记录。

### 4.4 `ref`
保存参考图的相对路径，比如：

- `ref/ref_01.png`
- `ref/ref_02.jpg`

之所以用**相对路径**，是为了让 case 自包含，不依赖某个人电脑上的绝对路径。

### 4.5 `check_points`
这是人工观察结果时重点检查的点，例如：

- 画风是否接近
- 人物服装是否正确
- 人数是否符合约束
- 动作是否稳定
- 画面是否出现明显崩坏

它的作用不是替代人工判断，而是让人工判断更一致。

### 4.6 `pass_rule`
这是通过规则，例如：

- 3 个检查点满足 2 个及以上算通过
- 人物数量必须完全正确，否则不通过
- 主角色风格接近且没有明显结构错误算通过

它负责把“感觉差不多”变成“至少有个口径”。

## 5. 这个工具解决的核心问题

这个项目主要解决的是**case 制作阶段**的问题，而不是评测全流程的一切问题。

它重点处理的是：

### 5.1 case 编辑
用户可以直接在网页里：

- 新建 case
- 编辑字段
- 增删 prompt
- 维护维度与子维度
- 填写检查点和通过规则

### 5.2 参考图管理
用户上传图片后，工具会自动：

- 统一放进 `ref/` 目录
- 自动命名为 `ref/ref_XX.ext`
- 把相对路径写回 `case.json`

这能避免图片散落各处、路径混乱的问题。

### 5.3 数据集导入导出
工具支持：

- 导出单条 case ZIP
- 导出完整数据集 ZIP
- 导入已有数据集 ZIP 继续编辑

所以它既能从零开始建库，也能在已有数据基础上接着整理。

### 5.4 浏览器本地保存
项目会把编辑状态保存在浏览器本地，减少“写到一半刷新没了”的低级事故。

### 5.5 搜索与筛选
当 case 逐渐变多时，可以按：

- 关键词搜索
- dimension 筛选

快速定位需要修改或复查的内容。

## 6. 它适合什么场景

这个工具尤其适合下面几类工作：

### 6.1 动漫文生图能力测试集制作
如果你在整理动漫风格图像生成模型的测试题库，它很顺手。

### 6.2 多人协作收集 case
一个人补 prompt，一个人挑参考图，一个人写检查点，也还能落在同一格式上。

### 6.3 早期评测体系搭建
很多团队一开始并不需要复杂平台，先把 case 结构整理起来，比急着上大系统更重要。

### 6.4 回归测试题库沉淀
当模型迭代后，可以持续补充“之前翻车过”的 case，逐步沉淀成回归集。

---

## 7. 它不是什么

为了避免预期跑偏，也要说清楚它**不**是什么：

- **不是模型推理平台**：不负责调用模型生成图片
- **不是自动打分系统**：不直接产出评测分数
- **不是训练数据管理系统**：重点是测试 case，不是训练样本治理
- **不是复杂工作流平台**：目标是轻量、清楚、够用

它做的是更前面的事情：

> 先把测试题目和题目结构整理干净。

这一步没做好，后面做自动化往往只是在放大混乱。

## 8. 当前功能概览

目前项目已经支持：

- 新建 / 复制 / 删除 case
- 编辑多值字段
- 维护预设维度与子维度
- 上传参考图并自动生成相对路径
- 实时预览 case 完整度
- 导出单个 case ZIP
- 导出整个数据集 ZIP
- 导入已有数据集 ZIP
- 浏览器本地自动保存
- 搜索与按维度过滤
- 明暗主题切换

整体设计思路很明确：

- 不做很重的后端
- 不引入复杂依赖
- 尽量保持静态网页即可部署
- 可以直接挂到 GitHub Pages 使用

---

## 9. 这个项目的设计取向

如果用一句话概括，这个项目的取向是：

> **轻量、结构先行、以 case 为中心。**

它不试图一口气解决测试平台的全部问题，而是先把最容易乱的一层收住：

- case 怎么定义
- 图片怎么挂
- 文件怎么导出
- 数据结构怎么统一

这其实很实用。
因为真实世界里，很多评测工作不是输在模型不行，而是输在：

- case 写得太散
- 标准不统一
- 文件结构乱
- 后续难复用

这个项目就是专门收拾这类混乱的。

---

## 10. 一句话总结

**Case Dataset Builder** 是一个用于整理动漫文生图测试 case 的轻量工具。

它把 prompt、参考图、检查点和通过规则统一收进结构化 case 里，并支持导入、编辑、导出，帮助团队更稳定地构建、维护和复用测试集。

如果你现在需要的不是“炫技平台”，而是一个**先把测试题库写清楚、整理干净、能稳定导出的工具**，那它就是干这个的。
